Identifiers for visually seperated components

Situation

Issue: There are many visual patterns in GCWeb that indicate a natural beginning and end to the content contained (aka has borders). However they are not represented equally when using a screen reader.

These include, but are not limited to:

  • alerts
  • panels
  • wells
  • cards (CDS design)
  • and/or pattern (when not using a list)
  • details (expand/collapse)

Other designs such as:

  • lists
  • blockquote
already read when moving in and out of the element.

Goal: Utilize a structure that can indicate three things:

  1. When you are entering the region
  2. What is the region
  3. When you are leaving (have left) the region

Proposal: Introduce both role="region" and aria-label or aria-labelledby when sectioning off content to better indicate when content begins and ends. Other aria or roles will be researched and considered.

Concerns:

It is recomended in the W3 org guide that roles and arias should be used sparingly (vague statement).

We need to consider if all patterns should get this design by default, and focus on creating less "call-out" designs, or only apply pattern to when the visual nature of the confined content is really important to get across.

JAWS only reads regions when set to high verbose (a level of granularity page structure feedback) or is set as a custom property. This means that some users may miss the content. We should consider either a fallback, or (preferred suggestion) ensure that the content within still adheres to proper code and structure semantic so it stands up even without a visual indicator (e.g. if the CSS styles did not load).

Testing platforms:

  • NVDA
  • JAWS 2025 (set to high verbose)
  • VoiceOver (macOS)

Details (ex/hide)

Details and summary elements do have an interactive element to showcase where to interact (except voiceOver, see below), but do not indicate where that content ends, unless the person already had navigated past prior to expanding.

Screen readers indicate that the content is a "button" that is collapsed when closed, and expanded when open. VoiceOver does not announce the details element as anything interactive when either tabbing or in read mode. This is a VoiceOver issue, but there might be some fallbacks if we decide to apply more accessibility attributes.
